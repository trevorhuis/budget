
# Budget Agent Guide

## Development environment

End-to-end local setup (prerequisites, `api/.env`, Postgres, migrations, dev servers, schema sync) lives in **`docs/DEVELOPMENT.md`**. Read that file before running installs, migrations, or both apps together.

## Current repo shape

- This checkout is in the middle of a flattening refactor from `apps/api` and `apps/web` to `api` and `web`.
- Prefer the current top-level `api/` and `web/` directories when making changes.
- Expect stale references to the old monorepo layout in deleted files, docs, and Docker config unless the user explicitly asks you to restore that structure.

## Workspace map

- `api/`: Hono API, Better Auth, Kysely, Postgres migrations, and end-to-end tests.
- `web/`: Vite app with React 19, TanStack Router, TanStack Query, TanStack DB, Tailwind 4, and shared UI primitives.
- Shared contracts come from the `schemas` package name. Both apps still link `node_modules/schemas` to `packages/schemas`, but that package is not present in this flattened checkout. Verify shared-schema availability before doing contract or build work.

## Runbook

- API dev server: `cd api && pnpm dev`
- API build: `cd api && pnpm build`
- API migrations: `cd api && pnpm migrate`
- API auth schema check: `cd api && pnpm auth:check-schema`
- API seed: `cd api && pnpm seed`
- API tests: `cd api && pnpm test:e2e`
- Web dev server: `cd web && pnpm dev`
- Web build: `cd web && pnpm build`
- Web lint: `cd web && pnpm lint`
- Local database only: `docker compose up postgres`
- Local backend plus database: `docker compose up`

## Backend rules

- API entrypoint is `api/src/index.ts`; all routes are mounted under `/api`.
- Keep the existing backend layering:
  - `api/src/api/*.router.ts`: HTTP routing, request validation, response mapping.
  - `api/src/core/<domain>/*.useCase.ts`: business logic and authorization decisions.
  - `api/src/core/<domain>/*.repository.ts`: database access.
  - `api/src/core/<domain>/*.model.ts`: zod schemas and domain types.
- User-scoped endpoints should continue to use `userMiddleware` plus `getAuthenticatedUserId`.
- Preserve the existing distinction between `401`, `403`, and `404` when touching protected resources.
- Database schema changes belong in numbered Kysely migrations under `api/src/db/migrations`.
- Tests rely on `api/src/test/test.utils.ts`, which drops or truncates tables in test mode. Do not point test runs at a non-test database.
- Chat and bulk transaction preview features require an OpenAI key. The backend accepts either `OPENAI_API_KEY` or legacy `OPEN_AI_API_KEY`. Bulk import can also use `OPENAI_BULK_TRANSACTIONS_MODEL`.

## Frontend rules

- Routing is file-based under `web/src/routes`.
- `web/src/routeTree.gen.ts` is generated by the TanStack Router plugin. Never edit it by hand.
- Route auth and onboarding guards live in `web/src/routes/_app.tsx`.
- Keep API transport concerns in `web/src/lib/api.ts`, including credentialed fetches and login redirects on `401`.
- Prefer going through TanStack DB collections in `web/src/lib/collections/*` instead of calling the API directly from components.
- Workspace pages are composed from `web/src/components/*-workspace.tsx`.
- Reuse the UI primitives in `web/src/components/ui/*` before adding new base components.
- Match the surrounding file style instead of applying broad formatting changes across the app.

## Known sharp edges

- `api/Dockerfile` still targets the old monorepo layout (`apps/api`, `packages/schemas`, `pnpm-workspace.yaml`) and does not match this flattened checkout.
- `web/README.md` still references `apps/web`, which is stale.
- The root `package.json` only declares the package manager. There is no root workspace script source of truth in the current checkout.
- In production, `web/src/lib/api.ts` requires `VITE_API_BASE_URL`. In development, `web/vite.config.ts` proxies `/api` to `http://localhost:3000` by default.
- API CORS currently allows `http://localhost:5173` and `https://*.vercel.app`.

## Editing guidance

- Make focused changes. This worktree already contains unrelated path-migration churn.
- Do not "fix" old-path references globally unless the task is specifically about finishing the monorepo-to-flat migration.
- If you touch routes, remember the route tree is generated by the Vite plugin during dev/build.
- If you touch shared contracts, expect both the API and web app to need coordinated updates.
